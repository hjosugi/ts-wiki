# Comprehensive Audit — 2026-07-06

A full multi-angle audit of ts-wiki, conducted with parallel deep-dives into
UX/design, search, customization/flexibility, server code quality, web code
quality, missing features, plus external research into Wiki.js (Requarks/wiki)
and five alternative wikis (BookStack, Outline, Docmost, Trilium, SilverBullet).

**Outcome: 218 issues filed (#82–#299)**, organized into the tracks below.

## Product direction set during this audit

- **Concept: `kawaii-wiki.ts`** (#261 rebrand) — shift the design language from
  corporate-docs minimalism toward pop/game-wiki/kawaii culture (lit.link as a
  customization reference), while making the product radically usable for
  non-engineers. Design vision RFC: #262.
- **VTuber-wiki enablement** — first-class embeds (YouTube/Twitch/X/pixiv/BOOTH),
  stream-schedule bridges to the existing event system, profile infoboxes, and a
  starter template set (#284–#293).
- **Frontend direction — DECIDED (2026-07-06):** keep the current **Lean Vue +
  UnoCSS** stack. Quasar and Nuxt migrations are not pursued; Nuxt is revisited
  only if SSR/SEO genuinely demands it later (SEO is covered cheaply by bot-meta
  injection #201 and sitemap #200). The RFC and spike issues (#294–#299) are
  closed. Component gaps stay in the current stack via the concrete issues
  (shared Modal #172, palette ARIA #156, dark mode #113, CSS-var theming #114).
  Research note that informed this: none of the surveyed wiki-class OSS uses Nuxt
  or Vue — Docmost is NestJS+React/Mantine, Outline is Koa+React.

## Implemented — v0.3.3 (2026-07-06 security/correctness batch)

- **#82 (fixed)** — Search now enforces per-page `page:read` ACLs: the
  `/api/search` route passes a `canRead` predicate into the search service, which
  over-fetches a bounded candidate window and drops pages the principal may not
  read, so denied titles/paths/snippets no longer leak. Regression test added.
- **#89 (fixed)** — `oidc.start()` sweeps expired `oauth_states` before inserting,
  so abandoned OIDC starts can't grow the table unbounded (mirrors the WebAuthn
  challenge cleanup).
- **#83 (closed, not a vulnerability)** — search snippets are already safe: the
  FTS `content` column is stored via `toPlainText()` (Markdown rendered with raw
  HTML disabled, then tag-stripped), so it is HTML-entity-encoded and the only
  live markup in a snippet is the `<mark>` highlight. Re-escaping would
  double-encode and regress display; a guard test locks in the safety property.
